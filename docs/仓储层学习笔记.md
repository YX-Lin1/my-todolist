# 仓储层（Repository）相关笔记

本文档整理自对 `library/db/main` 下各模块（以 `products`、`users` 为例）的问答，便于日后回顾。

---

## 1. `library/db/main/products` 目录里每个文件做什么？

| 文件 | 作用 |
|------|------|
| `products-repository.ts` | **接口（Port）**：约定「商品持久化」对外提供 `create` / `get` / `update`，不暴露 Drizzle 细节。 |
| `products-repository-impl.ts` | **实现（Adapter）**：用 `MainDbClient` + Drizzle 真正访问 `products` 表。 |
| `products-request.types.ts` | **入参类型**：调用仓储方法时传入的 `request` 形状。 |
| `products-response.types.ts` | **出参类型**：仓储返回给调用方的结果类型（如 `get` 可为 `null`）。 |

**为何这样拆？**

- 上层只依赖**接口**，便于测试（mock）、以后换实现。
- 请求/响应分文件，**进 / 出**一目了然，也方便以后响应单独演进。

**为何放在 `library/db/main/products`？**

- `library/db`：数据库访问集中放库内。
- `main`：主库这一上下文（与 `users`、`orders` 等并列）。
- `products`：按**领域/聚合**分文件夹，而不是把所有仓储堆在一个文件里。

---

## 2. `*-repository-impl.ts` 里的「实现」是什么意思？

- **实现的对象**：接口里定义的 `ProductsRepository` / `UsersRepository` 等。
- **要实现什么**：把「查 / 增 / 改」这些方法**真正连到数据库**（通过 Drizzle 和表定义），而不是只停留在类型约定上。
- **通俗类比**：接口像菜单上的菜名；实现像后厨真正做菜（用什么锅、哪张表、怎么写查询都在 impl 里）。
- **构造函数里的 `db`**：不自己造连接，由外部传入 `MainDbClient`，便于不同环境（测试/生产）注入不同客户端。

---

## 3. `create(...): Promise<UsersCreateResponse>` 是在「保证」返回值类型吗？

- **对 TypeScript**：算**编译期契约**——实现类必须满足该返回类型，调用方 `await` 后推断出的类型也是它。
- **对运行时**：**不保证**——类型编译后会被擦除，不会自动校验返回值形状；错误实现、`any`、未处理的异常仍可能导致实际行为与类型不一致。

结论：**这是类型层面的约定，不是运行时的强制校验。**

---

## 4. `*-request.types.ts` 是不是「数据库要求的传入格式」？

**不完全是。**

- 这些类型描述的是：**调用 `Repository` 的 `get` / `create` / `update` 时，参数对象应长什么样**。
- 表结构约束在 **Drizzle schema**；真正写入数据库的字段映射在 **`*-repository-impl.ts`** 的 `insert` / `update` 里。
- Request 类型常从 `$inferSelect` 推导，是为了和表**对齐、少重复**，但不等于「数据库 wire 格式」本身。

---

## 5. Request 和 Response 的「主体」是谁？（谁给谁？）

| 名称 | 方向 | 含义 |
|------|------|------|
| **Request** | 调用方 → 仓储 | 上层传给 `repo.get(request)` / `repo.create(request)` 等的参数。 |
| **Response** | 仓储 → 调用方 | 仓储 `return` 给上层的结果（如一行数据或 `null`）。 |

数据库在中间：仓储用 **Request** 里的数据去访问库，再把结果按 **Response** 类型交回调用方。

---

## 6. `UsersUpdateRequest` 里两行典型写法

```ts
data: Partial<UsersTableRow>;
id: UsersTableRow["id"];
```

- **`Partial<UsersTableRow>`**：行类型里每个字段都变成可选，表示**只传要改的字段**，不必带满整行。
- **`UsersTableRow["id"]`**：从行类型里取出 **`id` 列对应的类型**，表示「用主键定位要更新哪一条」。

合起来：**更新 `id` 这条记录，`data` 里只放需要变更的字段。**

---

## 7. `type UsersTableRow = typeof usersTable.$inferSelect` 是什么意思？

- **`usersTable`**：Drizzle 里定义的表对象。
- **`typeof usersTable`**：在类型层面取该表对象的类型。
- **`$inferSelect`**：Drizzle 提供的推断——**从这张表 `select` 出来的一行**在 TypeScript 里长什么样。
- **`type UsersTableRow = ...`**：给这个类型起别名，供 `UsersGetRequest`、`Partial<UsersTableRow>` 等复用。

**好处**：表结构主要在 schema 改一处，请求/响应相关类型更容易跟着变化，减少手写 DTO 与表不一致的问题。

---

## 8. 速查：常见符号

| 写法 | 含义 |
|------|------|
| `Promise<T>` | 异步；完成后得到 `T`。 |
| `Partial<T>` | `T` 的每个属性变为可选。 |
| `Omit<T, K>` | 从 `T` 里去掉属性 `K`。 |
| `T["id"]` | 索引类型：取 `T` 上 `id` 属性的类型。 |
| `typeof table.$inferSelect` | Drizzle：查询结果的行类型。 |
