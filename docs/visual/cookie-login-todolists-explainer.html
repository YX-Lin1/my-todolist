Cookie Auth

Cookie 的作用：让服务器“记住你是谁”

你现在的待办接口是 protectedProcedure，它只看服务端的 ctx.userId。 Cookie 就是把“当前登录用户”稳定地带到服务端，这样登录后查到的就是该用户自己的 todolist。

一句话理解 Cookie

Cookie 是什么

浏览器保存的一小段登录标识（比如 userId 或 sessionId）。

它做了什么

每次请求 /api/trpc，浏览器会自动带上 Cookie 给服务器。

为什么你需要它

服务器才能在 createTRPCContext(req) 里拿到 userId，进而按用户查待办。

登录到待办的完整链路（超简版）

Step 1

用户提交账号密码

登录接口验证成功后，拿到用户身份（通常是 userId）。

Step 2

服务器写入 Cookie

通过 Set-Cookie 保存登录态，建议用 HttpOnly。

Step 3

访问待办页发起 tRPC

浏览器请求时自动带 Cookie，不用你手动拼 userId。

Step 4

服务端识别并过滤数据

ctx.userId 注入到 Service，仅返回该用户的 todolist。

你项目里的“现在 vs 目标”

位置 现在 目标（Cookie 版）
登录页 只存 localStorage.loginToken 登录成功后触发服务端写 Cookie（推荐）
tRPC Context 只读 MOCK_USER_ID 优先从 req.cookie 解析 userId，开发可 fallback Mock
待办接口 protectedProcedure + ctx.userId（已就绪） 保持不变，直接复用
未登录处理 依赖 Mock，不明显 返回 401 后跳转 /login

最小实现步骤（按这个顺序）

  • 1) 登录成功后，在服务端设置 Cookie（Route Handler 或 Server Action）。
  • 2) 在 library/trpc/context.ts 中，从请求 Cookie 解析 userId。
  • 3) protectedProcedure 继续拦截未登录请求（你已经有了）。
  • 4) 去掉“依赖 Mock 才能看待办”的路径，保留开发 fallback 即可。

常用 Cookie 选项：HttpOnly、SameSite=Lax、Path=/、生产环境 Secure。 注意：仅把 token 放在 localStorage，服务器并不会自动知道你是谁。
